A learning atlas for experienced software engineers

Understand the machine around the model.

AgentDeck is a local control plane for coding agents. It starts external agent CLIs, gives them governed tools, preserves their history, and makes their work visible in a browser. This atlas explains the modern-AI concepts first, then the architecture that holds them together, then the Go choices that make the design practical.

Orientation

What AgentDeck is—and is not

It is not a chatbot UI calling a model API directly. It is a supervisor around existing coding-agent command-line tools such as Claude Code and Codex. The intelligence and many provider-specific behaviors live in those CLIs; AgentDeck owns the lifecycle, shared state, user controls, observability, and coordination surface.

Misleading mental model

“The browser sends a prompt to an LLM, then renders its answer.” That misses the agent process, tool calls, permission pauses, durable history, and all the failure modes around them.

Useful mental model

“The browser talks to a local supervisor. The supervisor runs agent processes, translates protocols, owns product state, and publishes a recoverable live view.” The model is one participant in a larger distributed system.

The one-page model

Three actors, four interfaces, one local source of truth

Think of AgentDeck as an orchestrator positioned between a human-facing web app and one or more agent CLIs. It makes different communication modes explicit instead of pretending everything is one request/response API.

Human surface

Browser UI

React views, REST commands, one SSE stream for live state, and a WebSocket only for raw terminal I/O.

↔
Control plane

Go binary

HTTP server, lifecycle supervisor, runtime registry, event bus, SQLite owner, and in-process MCP server.

↔
Agent execution

Agent CLIs

Claude/Codex-style processes that reason, call tools, manage native sessions, and emit structured or terminal output.

REST / commands and queries SSE / live projection ACP, MCP, hooks, PTY / agent integration
Boundary rule: the browser never talks directly to an agent CLI or to SQLite. That keeps agent supervision, security checks, state transitions, and recovery logic in one place: the Go server.

Priority 1 · AI development

Build around agents as stateful, tool-using processes

Modern AI-tool engineering is as much about orchestration, authority, observability, and lifecycle as it is about prompts or model selection. AgentDeck is a particularly concrete example because it treats the agent CLI as an external system with a protocol contract.

Highest priority

Core protocol

ACP: a programmable, bidirectional agent session

ACP is not a model API. It is a control protocol that turns an agent CLI into something a host application can start, prompt, observe, interrupt, resume, and answer permission requests for.

In AgentDeck
  • The chat runtime starts an ACP-compatible CLI and speaks JSON-RPC as newline-delimited JSON over stdin/stdout.
  • It initializes a session, sends prompts, receives streamed updates, and may receive an agent-originated permission request that needs a response.
  • It converts provider-shaped updates into AgentDeck events such as text, tool call, diff, permission request, turn end, and error.
Why it matters
  • Structured events are safer and richer than screen-scraping a terminal.
  • The host can own lifecycle, user approval, and durable history while the CLI remains responsible for its agent behavior.
  • ACP compatibility is still adapter- and version-specific; a protocol name does not erase integration testing.
Watch for the bidirectional part

In a normal HTTP API, the client asks and the server responds. ACP can reverse that direction: while handling a prompt, the agent asks the host for permission. AgentDeck pauses the request until the user approves, denies, times out, or cancels it. Treating this as merely a UI notification would let the agent continue before the decision reaches its execution boundary.

Tool protocol

MCP: capabilities exposed to an agent

MCP equips an agent; ACP controls an agent. MCP is the tool-server side of an agentic system: an agent discovers and calls capabilities that a host makes available.

In AgentDeck
  • The Go binary hosts one in-process, loopback HTTP MCP server at /mcp.
  • It gives agents a narrow coordination toolbox: list live peers, send mailbox messages, and read their own pending messages.
  • The MCP server is registered in an ACP session so an agent can use it while doing its work.
Design lesson
  • An MCP tool is an authorization surface, not just a convenience function.
  • AgentDeck binds a token to the caller’s server-side agent_id; it does not trust a sender identity supplied in a tool argument.
  • The tool server reads/writes the same state store as the rest of the product, avoiding a second messaging authority.
Current interoperability boundary

The shipped messaging transport is local streamable HTTP. Real CLI acceptance for HTTP MCP registration remains a credentialed gate, so the atlas distinguishes tested product wiring from verified behavior against every external CLI version. A hypothetical stdio proxy is not shipped.

MCP server message tools token registration live acceptance gates

Boundary design

Normalize once; adapt at the edge

External agent protocols are vendor-shaped; product behavior should be domain-shaped. Translate ACP updates and backend quirks at a single boundary before UI, persistence, and business logic consume them.

In AgentDeck

ACP mapping produces stable transcript events. A backend adapter contains variation in command invocation, environment cleanup, model handling, resume semantics, hooks, and permission behavior across Claude, Codex, OpenCode, and OpenHands.

Watch for

A common failure is assuming “all backends use ACP” means “all backends support every feature.” An interface reduces blast radius; it does not manufacture terminal hooks, resume support, or trustworthy model mapping where a CLI lacks them.

normalized event vocabulary backend adapter seam integration invariants

Context engineering

An agent run is more than a prompt

Context engineering is the discipline of deciding what an agent sees, when it sees it, and which inputs are allowed to change. It includes working directory, system instructions, project context, model selection, permissions, external tools, and native agent setup—not only the visible prompt box.

In AgentDeck

At launch, a project contributes its working directory and context; a role contributes its system prompt and policy; a backend/model contributes provider configuration; AgentDeck adds identity, hooks, and MCP registration. The composed input becomes a LaunchSpec.

Why freeze it

Running and resumed sessions must not silently change because somebody edited a role or project later. AgentDeck keeps a session snapshot for the high-level launch inputs, while launch-scoped credentials are rotated and native runtime behavior can remain provider-owned.

LaunchSpec launch composition session snapshot architecture flow

End-to-end trace

Follow one agent turn from click to durable history

This is the best single sequence to rehearse. Notice that the browser triggers work but is never responsible for keeping the agent alive or remembering what happened.

1

Launch is composed once

The server validates the requested role, project, backend, model, and interface; builds the launch snapshot; and mints launch-scoped artifacts such as hook/MCP tokens.

2

The runtime supervises the CLI

The registry selects chat or terminal runtime. Chat starts the ACP process and obtains an ephemeral native session identifier.

3

The user submits a prompt

The browser sends one REST command to the Go server. The runtime serializes the corresponding ACP write to the agent process.

4

Protocol updates become product events

Text chunks, tool activity, diffs, permission requests, turn completion, and errors are normalized into AgentDeck’s event vocabulary.

5

Durable state and live view diverge deliberately

State/transcript paths persist what needs recovery; the event bus fans a bounded, live projection to the browser over SSE.

6

The UI can reconnect and catch up

If an SSE event is missed, the UI hydrates current state and refetches durable transcript data rather than assuming perfect live delivery.

Multi-agent systems

Coordination is a durable mailbox plus a bounded wake-up loop

Agent-to-agent messaging is not model-to-model RPC. It is a product-owned mailbox: messages survive independently of a recipient’s current turn, and a nudger can ask an eligible idle chat agent to drain its mail.

In AgentDeck

Messages are stored in SQLite. A sender calls MCP send_message; the server records it, updates the recipient’s unread state, and signals the nudger. The recipient is prompted to call check_messages.

Control-loop lesson

Each agent turn has a combined message budget, and the nudger uses eligibility/cooldown rules. Those are guardrails against runaway feedback loops, not proof that natural-language messages are safe or that every wake-up will succeed.

durable mailbox nudger loop coordination design

Safety and reproducibility

Guard the execution boundary—and record where AI configuration came from

“Local” is a deployment choice, not a complete security model. AI supervision has two related jobs: authorize the actions an agent can take, and preserve enough redacted provenance to explain why it behaved as it did.

Permission & tokens

ACP permission requests pause the agent until a chosen response is relayed. Hook and MCP callers use launch-scoped tokens. The server binds to loopback and validates browser Host/Origin around all routes, including MCP and terminal paths.

Provenance & limits

Phase 7.5 ships read-only Claude/Codex source resolvers that report redacted effective values, source paths, fingerprints, and per-field provenance. Its watcher/API/launch integration remains planned for 7.6+, and it must never ingest secrets or create a second writable native-config authority.

Important product-boundary caveat

The dashboard API intentionally trusts same-machine callers, and child agent processes inherit the server environment except for adapter-specific stripping. That is a documented trade-off, not a guarantee that arbitrary local processes or unrelated credentials are isolated from an agent.

Priority 2 · Architecture

Design the control plane, then make the live view recoverable

AgentDeck is a local system, but it still has distributed-systems concerns: multiple processes, asynchronous event delivery, crashes, reconnects, mutable configuration, and external protocols. Its architecture works by assigning clear ownership to each concern.

Second priority

Control vs live data

REST expresses intent; SSE expresses what is happening

Long-running agent work should not be modeled as one giant HTTP response. A REST request begins or controls an operation; the live data plane communicates its evolving state.

In AgentDeck

REST is used for launch, prompt, stop, configuration, archive, and other finite operations. The multiplexed SSE stream carries live card updates, transcript events, notification events, and pings. The terminal is the narrow exception: raw interactive bytes use a WebSocket.

Why the separation helps

The server can keep supervising a process after a browser tab disappears. The UI can reconnect and rehydrate instead of polling every agent or holding the operation hostage to a single request.

route ownership SSE endpoint browser SSE client

Data ownership

Store data by its writer and access pattern—not by what it is “about”

Human-edited configuration, machine-generated state, and agent-owned transcript files have different owners. Giving each a distinct authority reduces accidental contention and makes the system explainable.

Config files

Roles, projects, backend settings, layout, and app settings are small JSON files: human-readable, hand-editable, and atomically written.

SQLite state

The Go server is the only writer of product state: agent identity, running registry, statuses, messages, session metadata, and full-text index.

Transcripts

Normalized NDJSON session history is append-oriented and indexed into SQLite for archive search. The index improves discovery; it is not a second truth source.

Stable identity is load-bearing: agent_id identifies the logical worker and survives stop, resume, model/backend/interface switch, archive, and mail. A CLI session_id, PID, and launch token describe one ephemeral incarnation.
storage rationale SQLite owner transcript writer FTS indexer

Event recovery

Live delivery may be lossy; recovery must be durable

A slow browser must not slow an agent. AgentDeck’s live bus uses bounded per-client buffers and can drop old events for a lagging consumer; the design works because clients can rebuild from current state and transcript reads.

The key pattern

When an SSE client connects, the bus captures a current state snapshot and subscribes the client atomically. This avoids the “snapshot first, subscribe second” hole where an update could fall between both actions.

The browser’s job

On reconnect, reset connection-scoped bookkeeping, hydrate current state, and refetch transcript data if per-agent event sequence numbers show a gap. SSE is a live projection—not the durable event log.

Why this matters beyond AgentDeck

This is a reusable streaming-system pattern: apply backpressure boundaries at the UI edge, preserve a canonical read path elsewhere, and make reconnection a first-class protocol. “Never drop an event” is often costlier and less robust than “detect, then catch up.”

event bus state projection rehydration logic

Ports and adapters

Keep the orchestration contract stable; put variability behind seams

The dashboard should not care whether a run uses ACP/stdio, a PTY, Claude, Codex, tmux, or iTerm. It speaks to narrow interfaces and normalized events; specialized components handle provider or driver differences.

Seams in this repo
  • Runtime handles lifecycle operations such as start, prompt, cancel, stop, resume, and subscribe.
  • The registry dispatches by agent interface, such as chat versus terminal.
  • BackendAdapter captures CLI-specific behavior; TerminalDriver captures xterm/PTY, tmux, or optional iTerm behavior.
Architecture trade-off

The interface contains change, but it also creates a contract checklist. A new runtime must join persistence, state updates, registration teardown, capabilities, security, and tests—not merely implement a Start method.

runtime contract runtime registry terminal driver seam backend adapter seam

Lifecycle architecture

Start, switch, crash, and stop are races—not simple functions

A reliable supervisor must know which launch generation owns a resource at every moment. External processes can exit late, commands can overlap, and old cleanup can accidentally destroy artifacts for a new run unless ownership is explicit.

In AgentDeck

The registry atomically claims launch/resume ownership; per-agent locks prevent overlapping switches; JSON-RPC writes are serialized; child CLIs run in process groups so Stop can signal the full process tree; crash paths remove registry ownership and launch-scoped registrations.

A useful distinction

Cancel asks an in-progress turn to stop while keeping the agent process alive. Stop tears down the process group and runtime ownership. Confusing the two produces either orphaned children or unnecessarily lost sessions.

atomic ownership process group supervision switch lifecycle lifecycle invariants

External configuration

Federation means reference, provenance, and one-way authority—not two-way sync

When a provider CLI owns its configuration, copying it into AgentDeck would create two writable sources of truth and an unsolvable merge story. The federation design reads native configuration while retaining AgentDeck’s ownership boundaries.

Linked

AgentDeck stores a binding and reads current native configuration. The native files remain authoritative.

Mirrored

Native configuration remains authoritative; AgentDeck may retain a redacted, disposable derived cache for compatibility.

Detached import

Selected non-secret values/assets become AgentDeck-owned and automatic sync stops. It is a deliberate ownership transfer.

Status matters: the typed schema and pure Claude/Codex resolver foundations are present in Phase 7.5. Watch/sweep, API/UI, freshness gates, and launch integration are planned in 7.6+. Do not read the design document as proof that every federation behavior is already live.

Reliability & security

Local-first reduces operations; it does not remove trust boundaries

Loopback binding helps, but browser-origin attacks, stale tokens, malformed protocol streams, and child-process access still need explicit handling.

Defenses in AgentDeck

The server binds to 127.0.0.1, applies Host/Origin validation across routes, gives hook/MCP producers per-launch tokens, and keeps state/transcripts/registration artifacts owner-readable only.

How it tests the boundary

A fake ACP executable lets tests exercise real process, HTTP, SSE, permission, and protocol paths without credentials. Targeted tests also cover malformed and large protocol frames, event backpressure, security guards, FTS variants, and UI behavior.

Priority 3 · Go

Go fits because AgentDeck is a small operating environment

The Go choice is not about a fashionable language preference. AgentDeck needs a compact local HTTP server, child-process supervisor, concurrent event router, SQLite owner, MCP host, and distributable binary. Go’s standard library and deployment model match that shape unusually well. The module currently declares Go 1.25.

Third priority

Why Go

One binary can own the whole local control plane

Go makes the supervisory layer boring to distribute. A compiled binary can serve HTTP, host MCP, manage processes, operate SQLite, and embed the built UI without asking end users to run a Node or Python service at runtime.

What Go provides here
  • net/http for REST, SSE, middleware, static assets, and local server lifetime.
  • os/exec and POSIX process controls for the external agent boundary.
  • embed to ship the compiled UI inside the binary.
  • Static types and explicit data shapes at protocol, state, and HTTP boundaries.
  • Packages—and the compiler-enforced internal/ boundary—to make ownership visible in the code layout.
Trade-offs

SQLite uses a cgo-backed driver in this project, so native build tooling still matters. Go also does not make concurrency safe automatically: goroutines are cheap, but shared maps, lifecycle state, and shutdown still need deliberate synchronization.

module dependencies embedded UI SQLite connection binary entry point

Concurrency

Goroutines do work; ownership rules make that work correct

Use goroutines for independent waiting, not as a substitute for a design. AgentDeck has readers for process output, request/response handling, SSE subscribers, background reconciliation, and shutdown work. Correctness comes from defining who owns each state transition.

Context

A context.Context carries cancellation/deadlines through a request or server lifetime. It tells cooperative work when to stop; it does not magically terminate a stubborn child process.

Mutex

A sync.Mutex protects critical shared state: pending RPCs, permissions, registry ownership, event snapshots, and write ordering. Keep the protected operation small and explicit.

Channel

A channel communicates or queues work. Bounded subscriber channels make backpressure visible. They do not remove the need for a mutex when multiple goroutines share a map or lifecycle invariant.

Good Go question: “Which goroutine owns this resource, and what happens if cancellation or a crash arrives between these two steps?” That question exposes most lifecycle bugs earlier than a clever abstraction does.
serialized protocol I/O ownership locking bounded event channels server cancellation

Systems programming

Process groups turn “stop the agent” into “stop the tree”

An agent CLI may spawn shells, language servers, or tool processes. Killing only the direct parent risks leaving children running after AgentDeck believes the session is gone.

In AgentDeck

Chat processes are started as a POSIX process group. A stop sequence signals the group, waits briefly, and escalates if needed. That makes lifecycle cleanup a system responsibility rather than a best-effort UI gesture.

Nuance worth learning

CommandContext and context cancellation are useful but are not a replacement for an explicit process-tree policy. Supervising external programs means handling graceful exit, timeouts, reaping, late exits, and stale PID records.

Go practices in the repo

Narrow interfaces, explicit errors, realistic fakes, and race checks

Go’s strongest contribution here is mechanical clarity. Packages expose small contracts, errors can be matched and mapped at the HTTP boundary, and tests can drive real protocol/process paths with a controlled fake agent.

Practices to notice
  • Interfaces such as Runtime and narrow persistence sinks mark real variation points, avoid import cycles, and make isolated tests possible.
  • Typed structs plus encoding/json keep public data shapes explicit; raw JSON is retained only where an external protocol requires flexibility.
  • Build tags exercise the shipped SQLite FTS5 path instead of assuming default tests cover it.
Verification mindset
  • Run normal tests plus the FTS5 tag variant for persistence/search changes.
  • Use the race detector around concurrent lifecycle or bus changes.
  • Test the real boundary with fakes: a fake ACP process catches framing, process, and event-flow bugs that a mocked method call cannot.
Two quiet Go lessons: filesystem durability and watcher skepticism

AgentDeck’s JSON configuration writes use a temp file, sync, rename, and directory sync because “rename is atomic” is not the full crash-durability story. Transcripts are append-only NDJSON and repair a crash-torn final record on reopening. Conversely, filesystem watcher events are treated as hints: transcript reconciliation also has a periodic fallback, while authenticated hook POSTs are the primary status path.

Mental-model checks

Six distinctions worth getting exactly right

These pairs are easy to blur at first. Keeping them separate will make the codebase and most modern agent-tool designs much easier to reason about.

Protocols

ACP ≠ MCP

ACP is the host-to-agent session contract; MCP is the agent-to-tool-server contract. AgentDeck composes them: it controls an agent over ACP and provides it a messaging toolset over MCP.

Persistence

Config ≠ machine state

Config is a human-owned declaration of future behavior; state is a server-owned record of what is or was true. Editing config should not rewrite an active session’s history or silently change its frozen launch inputs.

Identity

agent_id ≠ CLI session_id

The logical worker survives; a provider session does not. This is why AgentDeck can preserve its card, archive, messages, and history while relaunching on a different backend or interface.

Safety

Permission policy ≠ sandbox

Approve/deny governs one protocol execution point; it does not create OS-level isolation. Environment inheritance, filesystem access, network access, and tool permissions remain part of the larger trust model.

Streaming

SSE ≠ durable truth

SSE is the live projection; state and transcripts are the recovery paths. A reconnecting UI should hydrate and refetch when needed, rather than assuming every message arrived in order.

Go

Channel ≠ ownership model

A channel moves data or signals work; it does not by itself define who may mutate shared state. AgentDeck combines channels with mutexes, atomically claimed lifecycle state, and context-aware teardown.

Glossary

Fast definitions for the terms you will meet most

Search the atlas when you need a refresher. Each definition is deliberately tied to how the term changes the engineering problem in AgentDeck.

Search filters the concept cards and glossary terms on this page.
ACP — Agent Client Protocol
A bidirectional protocol for a host application to run and supervise an agent session. AgentDeck uses it over stdio for chat runtimes.
MCP — Model Context Protocol
A protocol for exposing tools/capabilities to an agent. AgentDeck’s in-process MCP server supplies coordination tools.
JSON-RPC over NDJSON
Request/response and notification messages encoded as one JSON object per line on a byte stream. ACP uses this style in AgentDeck’s chat runtime.
Launch snapshot
The frozen, redacted description of the high-level inputs used to start a session. It protects a resume from later configuration edits.
Provenance
Evidence of which configuration layer supplied a value and which source inputs were used. It is reproducibility for mutable AI configuration.
Source of truth / projection
The authoritative record versus a derived, replaceable view. SQLite/transcripts are durable sources; browser stores and search indexes are projections.
SSE — Server-Sent Events
A simple persistent HTTP stream from server to browser. It is ideal for live UI updates but needs reconnect and catch-up behavior.
PTY — pseudo-terminal
An OS interface that lets a program behave as if it is attached to a terminal. AgentDeck uses it for its terminal runtime.
Adapter / seam
A boundary that hides provider- or platform-specific variation behind a stable internal contract, such as BackendAdapter or TerminalDriver.
Capability token
A secret bearer value that grants a narrow right. AgentDeck maps launch-scoped MCP/hook tokens to an agent identity server-side.
Goroutine / channel / mutex / context
Go’s basic concurrency vocabulary: independent work, communication, shared-state protection, and cancellation/deadline propagation.
FTS5
SQLite’s full-text-search engine. AgentDeck indexes transcript-derived text so archived sessions can be searched without scanning every file.
