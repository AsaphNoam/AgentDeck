Local-first AI agent command center

Learn AgentDeck as a working tool, then as a machine.

AgentDeck is a single local Go binary and browser UI for launching, supervising, coordinating, switching, and resuming coding agents. This guide is written for a strong SWE ramping into Go and AI-tooling internals: first what to use at work, then what the product really built, then where to debug it.

What problem it solves at work

AgentDeck is for the moment when one AI agent is no longer enough: an implementer is coding, a reviewer is checking, a researcher is digging through history, and you need one local cockpit instead of a pile of terminal tabs and lost context.

Three concrete workflows
Parallel build + reviewLaunch implementer@repo, then a reviewer@repo. Let agents message each other through MCP instead of manually relaying summaries.
Bug triage across contextUse roles and projects to run a focused investigator while you keep the dashboard open for status, permissions, and commands.
Resume yesterday's workSearch Archive, inspect files/commands/transcript, then resume the same stable agent_id rather than starting from memory.
Switch when the tool fit changesMove a logical agent between model/backend/interface when needed, with native resume when possible and primer handoff when not.
What AgentDeck is not

It is not an IDE, not a cloud agent service, not a multi-user server, and not a complete terminal multiplexer replacement. It orchestrates local CLI agents, persists local state, and gives you one dashboard for supervision and recovery.

The basic operating loop

Configure roles, projects, and backends. Launch an agent from UI or CLI. Watch live state. Intervene through chat or terminal. Audit files, commands, messages, and assistant/tool transcript history. Note: user prompt bubbles are currently UI-local and are not yet persisted/searchable as transcript events.

Beginner mental model

The four ideas that make the repo legible

Read this before the protocol names. Most implementation details are variations on identity, storage, runtime interface, and communication plane.

Source: MAP.md
role@project is the human launch address A role supplies persona and permission policy; a project supplies cwd and context. Backend/model/interface choose the engine shape.
  • Example: implementer@agentdeck --backend claude --model sonnet
agent_id is product identity; session_id is runtime identity The card, archive, messages, group, and switch flow follow agent_id. Chat runtimes may carry provider session IDs; terminal launches can instead record tty/driver metadata.
Three stores, three jobs JSON config is human-editable. state.db is server-owned machine truth. sessions/{agent_id}/transcript.ndjson is the append log indexed for archive/search.
Two runtimes, one registry Chat is structured ACP over stdio. Terminal is a real CLI under PTY/xterm or tmux. The registry exposes one control surface to the server.
Switch is a controlled relaunch Switch-runtime stops the old runtime and resumes/relaunches the same agent_id. Same-backend paths prefer native resume; cross-backend paths use a bounded primer handoff.
Communication planes stay separate REST asks for actions. SSE pushes dashboard state. WebSocket carries terminal bytes. ACP carries structured chat. MCP is tool-calling between agents. Hooks report lifecycle/status.

Architecture flow Architecture decisions MAP.md

Big-picture moving-parts graph

The whole product is a browser UI, one loopback Go binary, one child process per agent, and local disk. REST commands move left-to-right; live state returns through SSE; terminal bytes use WebSocket; agent coordination uses MCP.

Mermaid-inspired · static SVG
How to read it

Start at the browser on the left. User actions enter the Go server through REST. The registry starts either chat or terminal runtimes. Agent output and hooks flow back into server-owned state, then return to the browser as SSE or terminal WebSocket bytes.

Control path: REST, registry, ACP
Live/state path: SSE, hooks, SQLite
Terminal path: WebSocket, PTY
Coordination path: MCP, nudger
The three big takeaways

AgentDeck is local-first: browser and agent CLIs never write SQLite directly; they talk to the Go server. The server is the control point that launches runtimes, receives status, writes state, and fans updates out. Chat content is structured ACP; terminal content is raw PTY bytes; both still share the same stable agent identity.

What is intentionally separate

Terminal bytes are not SSE messages, and chat transcript events are not hook posts. Config JSON is read at launch time, while state.db is live machine truth. That separation is what keeps switching, resume, archive, and dashboard updates understandable.

Identity graph: one logical agent, many runtime sessions

The most important invariant: every resume or switch keeps the same agent_id. Only the provider/runtime session_id changes.

Data/control planes

REST asks the system to do things. SSE tells the browser what changed. WebSocket carries terminal bytes. SQLite is written only by the server process.

ControlREST: launch, prompt, stop, resume, switch, config CRUD
Live pushSSE: state_update, new_message, notification, ping
TerminalWebSocket: browser keystrokes/resizes ⇄ PTY output
StateSQLite: identity, running, status, messages, sessions, FTS5
Control plane
  • REST launches, prompts, stops, resumes, switches, edits config.
  • Terminal WebSocket carries PTY bytes.
  • MCP tool calls coordinate agent-to-agent messages.
Live data plane
  • SSE pushes state_update, new_message, notification, ping.
  • UI stores hydrate and update from the stream.
  • No browser polling of agent CLIs.
Persistence plane
  • Config is JSON under ~/.agentdeck.
  • Machine state is SQLite, server-only writer.
  • Transcripts are append logs indexed into FTS5.

Debugging heuristic

If the UI looks stale, inspect SSE and stores. If launch fails, inspect config composition and backend adapter. If archive/search is wrong, inspect transcript writer and indexer. If a switch loses identity, inspect agent_id vs session_id.

Feature to package map

Routes: internal/server/routes.go ACP chat: internal/runtime/chat.go Terminal: internal/runtime/terminal State: internal/state/schema.go MCP tools: internal/messaging/tools.go Hooks: internal/hooks/hooks.go Indexer: internal/index/indexer.go UI live stream: ui/src/api/sse.ts

Human-facing config is readable JSON. Machine-facing state is SQLite because AgentDeck needs identity joins, running/session status, message rows, and FTS5 search.

Hands-on labs

Small loops that teach the product

Each lab is deliberately tiny: do one thing, observe one result, then inspect the package or file that owns it.

Source: README quickstart
1. Start the dashboardRun make dist, start/open the dashboard, then inspect ~/.agentdeck.
  • Expected: seeded roles/projects/backends plus state.db.
  • Read: internal/config, README.md.
2. Configure and launch one agentCreate or edit a role/project/backend, then launch from the modal or CLI.
  • Expected: one card with a stable agent_id.
  • Read: internal/server/launch.go.
3. Trace one chat turnSend a prompt and follow UI -> REST -> ACP -> normalized event -> transcript/SSE.
  • Expected: assistant/tool events stream; your prompt bubble is not durable yet.
  • Read: chat.go, acpmap.go, writer.go.
4. Send an agent messageAsk one agent to use send_message to another live chat agent.
  • Expected: message row, unread badge, possible nudger turn.
  • Read: internal/messaging.
5. Search and resumeStop an agent, search Archive for phrase/file/command context, then resume it.
  • Expected: same agent_id, fresh runtime, prior indexed history.
  • Read: internal/archive, internal/server/resume.go.
6. Switch runtime or modelUse the card context menu to switch backend/model/interface and verify identity stability.
  • Expected: same card identity; native resume or primer handoff depending on target.
  • Read: internal/server/switch.go.
Stale cardCheck /api/events, internal/bus, status row writes, and the Zustand agent store.
Prompt missing from ArchiveExpected today: user prompt bubbles are client-local. Search assistant/tool output instead until user_text persistence lands.
Blank terminalCheck /api/capabilities, selected driver, terminal WebSocket, PTY hub/scrollback, and hook status separately.
Message not deliveredCheck MCP token registration, recipient resolution, chat-vs-terminal recipient rules, unread rows, and turn budget.
Switch failed or lost historyCheck switch lock, resume ID resolution, primer generation, rollback path, and the frozen session snapshot.

Current status

Current Truth: live, gated, sharp edges

Source: HANDOFF.md

Green today

  • Phases 0-5 are complete: foundation, core ACP chat loop, dashboard/state, config/onboarding, archive/search/resume, MCP coordination, nudger, budgets, notifications.
  • Phase 6 subphases 6.1-6.6 landed: backend adapters, hook registration, xterm/tmux terminal runtime, switch-runtime, backend primer handoff, task groups, and UI endpoints.
  • Terminal PTY has in-progress uncommitted hub work in this tree: one reader drains the PTY, broadcasts to subscribers, and replays bounded scrollback if that patch is kept and green.
Still gated or optional

Codex ACP wiring exists but live Codex chat acceptance is credential-gated. Codex hook names and live two-CLI HTTP MCP registration still need real CLI validation. Phase 6.7 iTerm2/AppleScript is optional. Phase 7 activity map remains future polish.

Open blockers to respectHANDOFF currently tracks onboarding backend clobber, primer persistence during switch, cancel escalation crossing turns, and permission resolution races. Treat these as product truth until fixed.
Not persisted yetUser prompt bubbles are not durable transcript/search records. Assistant/tool events, files, commands, messages, session metadata, and FTS rows are the durable surfaces today.
Terminal nuanceRaw terminal bytes are not the same as normalized ACP transcript events. The PTY path is capability-gated, hook-driven for status, and its multi-view/no-viewer behavior depends on the current PTY hub patch being completed and verified.
MCP transport realityAgentDeck mounts streamable HTTP MCP at /mcp and binds caller identity through generated per-agent registration. A working stdio proxy fallback is not currently shipped.
Static AgentDeck atlas. One file, no external assets, no backend calls.
